# 制造业八大核心系统平台（MVP）
## 产品需求文档（PRD）v1.1

**版本**：v1.1（MVP全功能版）  
**日期**：2026-04-14  
**团队**：3人全栈（Node.js）  
**目标行业**：线缆制造（切入点），可扩展至离散制造全行业  

---

## 1. 产品概述

### 1.1 产品定位

**"制造转换引擎"驱动的可配置制造平台**

以"物料形态转换"为通用内核，八大系统（PLM/SCM/ERP/APS/MES/WMS/QMS/EAM）围绕转换过程各司其职，支持单系统售卖或任意组合，通过适配器对接外部系统。

**MVP原则**：
- **功能完整**：每个系统都是完整功能版本，非简化版
- **体验做好**：界面、交互、性能不妥协
- **范围聚焦**：先实现核心闭环，但闭环内功能完整
- **质量优先**：宁可少做，也要做好

### 1.2 核心价值主张

| 客户痛点 | 我们的方案 |
|:---|:---|
| 信息化0基础，不知从何开始 | 模块化起步，先上核心，逐步扩展 |
| 担心被单一系统绑定，后期无法扩展 | 统一内核，八大系统无缝衔接 |
| 已有部分系统，怕重复投资 | 适配器对接，保护现有投资 |
| 行业特殊，标准产品不适用 | 元数据配置，不换系统换配置 |

### 1.3 目标用户

**第一种子客户**：信息化0基础的中小型线缆制造企业（50-200人，年产值3000万-2亿）

**用户画像**：
- 生产副总/厂长：关注交期、质量、成本
- 质量主管：关注追溯、验厂报告
- 仓库主管：关注库存准确、找料效率
- 老板：关注投入产出、避免大而全的失败

│  单点归属：每个系统有明确的Owner数据，组合时数据衔接       │
│  转换引擎：物料形态转换的通用模型（输入→过程→输出）        │

### 2.2 转换引擎内核（核心抽象）

**通用模型：物料形态转换**

```
转换定义（Conversion Definition）
├── 转换ID
├── 输入物料清单（物料ID + 数量 + 批次要求）
├── 输出物料清单（物料ID + 产出比例 + 是否主产出）
├── 资源需求（设备类型、工时、人力、能源）
├── 工艺参数（温度、压力、速度等，可扩展）
├── 质量要求（检验项、标准值、判定规则）
└── 时间参数（标准工时、准备时间、 teardown时间）

转换实例（Conversion Instance）
├── 实例ID（对应工单/订单/批次）
├── 引用转换定义（模板）
├── 实际输入（物料批次追溯链）
├── 实际输出（产出批次、数量、质量状态）
├── 实际资源消耗（设备ID、操作工、实际工时、能耗）
├── 质量记录（检验数据、不合格处理、放行记录）
├── 时间轴（计划开始/结束、实际开始/结束、中断记录）
├── 位置轨迹（工位、线边仓、主仓移动记录）
└── 成本归集（料工费分摊到产出）
```

**五大关注统一归集**：
- **料**：输入输出物料平衡、批次追溯链
- **成本**：资源消耗归集到转换实例，分摊到产出
- **质量**：检验标准、执行记录、不合格处理
- **交期**：转换实例的时间轴、关键路径
- **效率**：资源利用率、OEE、产出率

### 2.3 八大系统职责边界（单点归属）

| 系统 | Owner数据 | 核心闭环 | 产出服务 | 消费服务 |
|:---|:---|:---|:---|:---|
| **PLM** | 物料主数据、BOM、工艺路线、变更版本 | 设计源头管理 | 物料/BOM/工艺 | 质量反馈、成本分析 |
| **SCM** | 供应商档案、采购订单、到货计划 | 采购闭环 | 物料供应、供应商评价 | 库存预警、生产需求 |
| **ERP** | 财务科目、销售订单、成本核算、总账 | 财务业务一体化 | 销售需求、财务凭证 | 生产完工、采购实绩、库存变动 |
| **APS** | 产能模型、排程规则、优化算法 | 计划优化 | 派工单、交期承诺 | 订单池、资源状态、工艺路线 |
| **MES** | 生产工单状态、在制品、工位实绩 | 生产执行 | 完工实绩、消耗记录、追溯数据 | 工艺路线、生产计划、物料可用、设备状态 |
| **WMS** | 库存实物、库位、批次、出入库记录 | 仓储作业 | 库存变动、物料可用 | 采购到货、销售发货、生产领退 |
| **QMS** | 检验标准、不合格品、质量档案、SPC | 质量管控 | 质检报告、放行记录、质量追溯 | 工艺要求、生产批次、客户投诉 |
| **EAM** | 设备台账、维保计划、故障记录、备件 | 设备全生命周期 | 设备状态、OEE、维修历史 | 生产计划（设备需求）、故障反馈 |

### 2.4 系统间衔接（MVP必须实现）

```
数据流向：

PLM ──物料/BOM/工艺──────────→ 所有系统（基础数据）
                                ↓
SCM ──采购到货───────────────→ WMS ──物料可用──────→ MES/APS
                                ↑
ERP ──销售订单───────────────→ APS ──派工单────────→ MES
                                ↓                   ↓
                                └────────成本实绩───→ ERP

EAM ──设备状态/OEE───────────→ APS（排程约束）
      设备状态──────────────→ MES（工位可用校验）
                                ↓
QMS ──检验标准───────────────→ MES（执行依据）
      ←────────质量记录────── MES（执行反馈）
```

**8个关键衔接点**：

| 序号 | 衔接 | 数据内容 | 触发方式 | 一致性要求 |
|:---:|:---|:---|:---|:---|
| 1 | PLM→全系统 | 物料/BOM/工艺变更 | 变更推送 | 最终一致 |
| 2 | SCM→WMS | 采购到货通知 | 到货确认 | 实时 |
| 3 | WMS→MES | 物料可用量、领料出库 | 实时查询/事件推送 | 实时 |
| 4 | ERP→APS | 销售订单、优先级 | 订单下达 | 实时 |
| 5 | APS→MES | 派工单、工艺路线 | 排程完成推送 | 实时 |
| 6 | MES→ERP | 完工实绩、成本归集 | 完工触发 | 准实时（<5分钟）|
| 7 | EAM→APS/MES | 设备状态、OEE、维保计划 | 状态变更推送 | 实时 |
| 8 | QMS↔MES | 检验标准、质量记录 | 检验节点触发 | 实时 |

## 4. 适配器框架（Connector）

### 4.1 设计目标
- 内核纯净：八大系统内部只用标准模型，不感知外部系统
- 快速适配：新外部系统通过配置+少量脚本实现对接
- 双向可选：支持只接下行（取数）、只接上行（回传）、或双向
- 降级可用：外部系统故障时，内部继续运转，数据补录

### 4.2 架构

```
┌─────────────────────────────────────────┐
│  适配器引擎（统一框架）                    │
│  协议适配层：REST / SOAP / 数据库 / 文件 / MQ │
│  数据映射层：字段映射 / 编码转换 / 单位转换   │
│  调度层：定时 / 实时 / 批量 / 事件驱动       │
│  异常处理：重试 / 告警 / 人工干预 / 死信队列   │
│  监控层：对接状态 / 数据量 / 延迟 / 错误率     │
└─────────────────────────────────────────┘
              ↑
    ┌─────────┴─────────┐
    ↓                   ↓
┌─────────┐        ┌─────────┐
│ 适配器配置 │        │ 运行时监控 │
│ （YAML/界面）│        │ （可视化监控台）│
└─────────┘        └─────────┘
```

### 4.3 MVP适配器清单

| 优先级 | 适配器 | 对接系统 | 数据流 | 场景 |
|:---:|:---|:---|:---|:---|
| P0 | 金蝶K3Cloud | ERP | 双向 | 客户已有金蝶，要我们的生产系统 |
| P0 | 用友U8/U9 | ERP | 双向 | 同上 |
| P1 | 鼎捷 | ERP | 双向 | 台湾企业常用 |
| P1 | 第三方MES | MES | 单向（取数） | 客户已有MES，要我们的质量/仓储 |
| P1 | 电商平台 | 淘宝/京东/拼多多 | 单向（取数） | 电商接单，内部生产 |
| P2 | 供应商门户 | 供应商系统 | 双向 | 采购协同 |
| P2 | 物流平台 | 顺丰/德邦等 | 双向 | 物流跟踪 |

### 4.4 配置示例（金蝶对接）

```yaml
connector:
  id: "kingdee_k3cloud"
  name: "金蝶K3Cloud"
  protocol: "rest"
  endpoint: "https://client-k3cloud.com/api"
  auth:
    type: "oauth2"
    client_id: "${CLIENT_ID}"
    client_secret: "${CLIENT_SECRET}"

  # 下行：从金蝶获取数据
  inbound:
    - name: "销售订单"
      source: "T_SAL_ORDER"
      target: "erp_sales_order"
      mapping:
        - field: "FBillNo" → "order_no"
        - field: "FDate" → "order_date"
        - field: "FCustId.FNumber" → "customer_code" 
          transform: "lookup:customer_mapping"
        - field: "FMaterialId.FNumber" → "material_code"
          transform: "lookup:material_mapping"
        - field: "FQty" → "quantity"
        - field: "FDeliveryDate" → "delivery_date"
      filter: "FDocumentStatus = 'C'"  # 已审核
      schedule: "0 */5 * * * ?"  # 每5分钟

    - name: "库存余额"
      source: "T_STK_INVENTORY"
      target: "erp_inventory"
      mapping: ...
      schedule: "0 0 * * * ?"  # 每小时

  # 上行：回传金蝶
  outbound:
    - name: "生产入库单"
      source: "mes_finish_record"
      target: "T_PRD_INSTOCK"
      mapping:
        - field: "work_order_no" → "FSrcEntryID"
        - field: "material_code" → "FMaterialId.FNumber"
          transform: "lookup:reverse_material"
        - field: "finish_qty" → "FRealQty"
        - field: "finish_time" → "FDate"
        - field: "warehouse_code" → "FStockId.FNumber"
      trigger: "event:mes_workorder_finish"

    - name: "采购应付"
      source: "scm_purchase_receive"
      target: "T_AP_PAYABLE"
      mapping: ...
      trigger: "event:scm_receive_approve"

  # 异常处理
  error_handling:
    retry: 3
    retry_interval: 60  # 秒
    alert: "webhook:dingtalk"
    dead_letter_queue: true
    manual_retry_ui: true

  # 监控
  monitoring:
    enabled: true
    metrics: ["request_count", "error_rate", "latency", "data_volume"]
    alert_threshold:
      error_rate: 0.05  # 5%
      latency: 5000     # 5秒
```
