## 一、问题分类标准

| 分类 | 说明 |
|:---:|:----|
| ✅ | 完全匹配 — 前端列名在后端实体中有对应的字段名（包括关联表查询和计算字段） |
| ⚠️ B类 | 命名差异 — 前端列名与后端字段名不一致但含义相同（如 `supplierName` vs `supplierId`） |
| ❌ A类 | 字段不存在 — 前端列名在后端实体中找不到对应字段，可能导致运行时错误 |

### A-5. PLM → code-rule → `prefix`

| 位置 | 文件 | 行号 | 代码/片段 |
|:----|:----|:----:|:---------|
| 🔴 前端列定义 | `frontend/src/views/plm/code-rule/index.vue` | 94 | `{ key: 'prefix', title: '前缀', dataIndex: 'prefix', width: 100 }` |
| 🔴 前端表单字段 | 同上 | 139 | `{ field: 'prefix', label: '前缀', type: 'input', ... }` |
| 📗 后端实体 | `backend/src/modules/plm/entities/plm-material-code-rule.entity.ts` | 21-66 | 字段列表: `id`, `tenantId`, `code`, `name`, `codeType`, `categoryId`, `segments`(JSON), `serialLength`, `currentSerial`, `separator`, `isDefault`, `status`, `createdBy`, `createdAt`, `updatedAt` |
| 📗 实体 JSON 结构 | 同上 | 10-16 | `segments` 是 `CodeSegment[]`，其中 `type: 'PREFIX'` 的项包含前缀信息，不作为独立列 |

**分析**: 实体的编码规则通过 `segments` (JSON) 存储各段规则，前端将其展开为 `prefix`, `format`, `serialLength` 等独立字段。

**建议**: 
- 方案A: 实体补充 `prefix` 列
- 方案B: 前端从 `segments` JSON 中提取 PREFIX 段展示

| 位置 | 文件 | 行号 | 代码/片段 |

**分析**: 前端 `seqDigits` ↔ 实体 `serialLength`，语义一致但命名不同，可归为 B 类。但如果代码中直接使用 `item.seqDigits` 会失败。

**建议**: 统一命名或添加 getter/setter。
